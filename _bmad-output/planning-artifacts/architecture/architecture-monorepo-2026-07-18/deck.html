Architecture Spine · MVP1

Un Shell, tres Remotes, ninguna dependencia cruzada.

Llano es un monorepo Nx que federa en runtime una app Shell y tres microfrontends Angular independientes —Catálogo, Carrito/Checkout, Perfil— vía Native Federation. Este documento recorre las decisiones que mantienen esa independencia real: qué puede depender de qué, cómo se comparte estado entre Remotes sin acoplarlos, y qué pasa cuando uno falla.

Paradigma: micro-frontend / shell + remotes Federación: Native Federation, estática (MVP1) Nx 23.1 Angular 22

01 — Contexto

Por qué existe este proyecto

Es un ejercicio de aprendizaje construido para entender —de primera mano, no en teoría— cómo se coordinan builds, dependencias compartidas y despliegue entre aplicaciones Angular independientes en un mismo repositorio. No hay backend real, ni pagos, ni usuarios reales: los datos son mock. El valor está en la arquitectura misma, y en poder explicar con propiedad cada decisión detrás de ella frente a un recruiter o entrevistador técnico.

Fuente: PRD §1 Vision — prd-monorepo-2026-07-17.

02 — Design Paradigm

Micro-frontend: Shell + Remotes federados

El Shell carga los tres Remotes en tiempo de ejecución vía Native Federation, sobre el ApplicationBuilder (esbuild) de Angular — sin depender de Webpack. En MVP1 la federación es estática: las URLs de los tres Remotes están fijas en la configuración del Shell. Angular y RxJS se comparten como Singleton para que ninguno de los cuatro proyectos cargue su propia copia.

03 — Organización de libs

Tipado Nx estándar, no carpetas ad-hoc

Elegido explícitamente por valor de aprendizaje: demuestra el patrón real de Nx, no una SPA repartida en carpetas. Cada lib lleva dos tags — scope (a qué dominio pertenece) y type (qué rol cumple) — enforced con @nx/enforce-module-boundaries.

TipoRolPuede depender de
featureSmart components, rutas, orquestación de un dominioui, data-access, util del mismo scope; util/data-access de scope:shared
uiComponentes presentacionales puros, sin estado de negocioutil únicamente
data-accessEstado y acceso a datos del dominio (services, signals, mocks)util únicamente
utilTipos, helpers puros, sin Angular DI de negocionada — hoja del grafo

04 — Invariants & Rules

Las siete decisiones que evitan que dos fases del build diverjan

Cada una nombra qué liga (Binds), qué divergencia evita (Prevents) y la regla enforceable (Rule). Todas [ADOPTED] — ya decididas, no abiertas.

AD-1

Ningún Remote depende de otro Remote

Binds
FR-1, FR-9, los 3 Remotes
Prevents
Acoplamiento directo entre Remotes que rompería la independencia de build/serve de FR-9.
Rule
apps/catalogo, apps/carrito, apps/perfil solo dependen de libs/shared/*. Enforced vía tags scope:*.
AD-2

Dirección de dependencia por tipo de lib

Binds
Todas las libs del workspace
Prevents
Que ui llame data-access directo, o que util deje de ser una hoja segura.
Rule
feature → {ui, data-access, util} → util. Ningún salto hacia atrás. Enforced vía tags type:*.
AD-3

Estado cross-remote mutado solo vía servicio singleton + Signals

Binds
FR-4, FR-6, FR-7, libs/shared/cart, libs/shared/auth
Prevents
Que Catálogo y Carrito implementen cada uno su propia copia del carrito; que editar Perfil deje al Session Widget con un nombre desactualizado.
Rule
Un único service providedIn: 'root' por lib, con signal()/computed() como fuente de verdad. Editar Perfil llama al mismo método que usa el login.
AD-4

Tipos de entidades cross-remote en una lib neutral

Binds
FR-4, FR-6, FR-7, Product, CartItem, User
Prevents
Que Catálogo y Carrito terminen con formas ligeramente distintas del mismo Product.
Rule
libs/shared/util-types exporta las interfaces; price siempre number decimal, nunca centavos.
AD-5

Fallo de carga de un Remote se aísla en su propio outlet

Binds
FR-1, estado "Remote no disponible"
Prevents
Que un Remote caído tumbe la navegación completa, o que el error sea silencioso para un lector de pantalla.
Rule
loadRemoteModule().catch() → RemoteUnavailableComponent con role="alert". "Reintentar" reinvoca la carga, nunca un reload completo.
AD-6

Angular/RxJS y libs compartidas, Singleton en la federación

Binds
FR-1, FR-4, FR-7, NFR Reliability
Prevents
"Version mismatch" entre Shell y Remotes; que cada Remote bundlee su propia copia del service de carrito/sesión.
Rule
federation.config.mjs declara Angular, RxJS, auth, cart y util-types como singleton: true.
AD-7

Un solo deploy a Cloudflare Pages, base-href por Remote

Binds
FR-11, deployment
Prevents
Que el catch-all del Shell se trague los assets de un Remote; que un Remote se buildee con el base-href equivocado.
Rule
Cada Remote con --base-href=/{r}/. _redirects lista las reglas de cada Remote antes que el catch-all del Shell.

06 — Structural Seed

Un origin, cuatro apps, cero CORS

Shell y los tres Remotes se sirven desde el mismo origin de Cloudflare Pages, cada uno en su propio sub-path — deliberadamente MVP1-only: el PRD difiere el deploy independiente por Remote a una extensión futura.

09 — Qué demuestra este proyecto

Success metrics, no solo "funciona"

SM-1

Explicar y demostrar en voz alta cómo funciona Native Federation (host/remote/singleton) y por qué se prefirió sobre Module Federation clásico.

SM-2

Shell + 3 Remotes corren federados, local y en producción, sin errores de version mismatch entre Angular/RxJS.

SM-3

Demo pública desplegada, accesible vía URL compartible, navegable end-to-end (Catálogo → Carrito → Perfil).

Counter-metrics — qué NO optimizar

No copiar patrones de tutoriales sin entenderlos. No sacrificar la independencia real de cada Remote con atajos que los acoplen solo para ahorrar tiempo.

10 — Deferred

Lo que esta arquitectura deja abierto, a propósito

  • TestingProfundidad de unit/e2e cross-remote — abierto en el PRD (Open Question 4).
  • CI/CDDeploy independiente por Remote — extensión futura confirmada, fuera de MVP1.
  • MVP2Federación dinámica vía Manifest — dirección ya fijada, forma sin decidir.
  • EstadoEscalar a NgRx si el carrito/sesión creciera en complejidad — hoy Signals alcanza.
  • ResilienciaAislar errores de runtime dentro de un Remote ya montado (más allá de fallos de carga) — Angular no tiene error boundaries nativos por subárbol.
